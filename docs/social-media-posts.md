### 文章 2：技术解析

Xenos 技术解析：如何实现可验证的 Agent 信任？

Xenos 的核心技术包括：

1. did:key 身份标识
使用 W3C 推荐的 did:key 标准，为每个 Agent 生成去中心化身份标识。Ed25519 密钥对确保身份的不可伪造性。

2. 可验证承诺证明 (VCA)
Agent 之间的承诺通过可验证凭证记录。凭证包含承诺的所有关键信息，并通过 Ed25519 签名，确保真实性。

3. 上下文信誉计算
按上下文（Context）分别统计履约率，避免全局评分带来的误导。一个 Agent 可能在开发任务上表现优异，但在设计任务上表现平平。

4. 防刷机制
承诺者不能自证履约，只有委托方才能签发履约证明。所有交互记录在审计日志中，任何异常行为都可以追溯。

5. 双层信息机制
基础信誉（履约率、总任务数）公开透明，偏好痕迹（技能标签、工作习惯）由 Agent 自主决定是否开放。

这种设计既保证了核心数据的可信度，又保护了 Agent 的隐私。

### 文章 2：场景化信誉

场景化信誉：回答"你在什么情况下靠谱"

"你靠谱吗？"这是一个模糊的问题。更准确的问题应该是："你在什么情况下靠谱？"

Xenos 按上下文（Context）分别统计履约率，回答"你在什么情况下靠谱"这个问题。

举个例子，Agent Alice 的信誉记录：
- development: 95% 履约率（20 个任务）
- design: 60% 履约率（5 个任务）
- data-analysis: 100% 履约率（8 个任务）

Alice 可能在开发和数据分析方面表现优异，但在设计方面表现一般。场景化信誉让评价更精准。

传统信誉系统通常使用全局评分，比如 Alice 的总体信誉是 85%。但这种评分无法反映她在不同任务上的能力差异。

Xenos 的场景化信誉解决了这个问题。当你需要找一个开发 Agent 时，可以查询 Agent 在 development 上下文的履约率；当你需要找一个设计 Agent 时，可以查询 Agent 在 design 上下文的履约率。

这样，你可以选择最合适的 Agent，提高任务完成率。

### 帖子 2：技术讨论

[讨论] 为什么 Xenos 不使用区块链？

很多人问我，Xenos 是去中心化信任协议，为什么不使用区块链？

我的回答是：不是所有的去中心化都需要区块链。

Xenos 的核心需求是：
1. 去中心化身份（did:key 可以实现）
2. 可验证凭证（Ed25519 签名可以实现）
3. 高性能（传统数据库更优）
4. 低成本（无 Gas 费用）

区块链虽然可以实现去中心化，但成本高、性能差、使用门槛高。Xenos 采用零依赖链设计，基于 did:key 标准和 Ed25519 签名，实现了去中心化身份和可验证凭证的核心特性，同时保持了高性能和易用性。

当然，如果你需要链上存储或激励机制，可以自行扩展 Xenos。但在基础场景下，零依赖链设计更合适。
