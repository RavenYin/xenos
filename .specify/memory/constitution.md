# Xenos 项目宪法

## 项目定位

Xenos 是面向 AI Agent 的轻量级承诺证明系统。

## 核心理念

### 一、信任不应预设，而应可证

Agent 之间不应预设信任，而应通过可验证的履约记录建立信任。

- 每个承诺都有完整的生命周期记录
- 状态变更写入不可篡改的审计日志
- 履约需要证据支持（链接、PR、文档等）
- 验收结果形成可验证凭证

### 二、声誉不是全局评分，而是上下文绑定的履约记录

一个 Agent 在不同领域的信誉是独立的。

- 在"开发"领域履约 95% ≠ 在"设计"领域履约 95%
- 查询信誉必须指定上下文：`GET /reputation?userId=xxx&context=development`
- 不同上下文的信誉不叠加、不平均
- 每个上下文独立计算履约率

**为什么上下文绑定？**
> 一个优秀的程序员，未必是优秀的设计师。全局评分会掩盖这种差异，让信任决策失真。

### 三、每个承诺都是一份可验证的"数字契约"

承诺不是口头约定，而是可追溯、可验证的数字凭证。

- 承诺创建时记录：谁承诺、承诺什么、给谁、截止时间
- 履约时提交证据：GitHub PR、文档链接、截图等
- 验收时生成凭证：通过/不通过 + 评语
- 所有操作可追溯：审计日志完整记录

## 轻量原则

- 核心 API 仅需 HTTP + JSON
- 无区块链依赖（纯链下实现）
- SDK 一行安装：`npm install @xenos/vca-sdk`
- 响应时间 < 500ms

## 开放原则

- 所有端点支持程序化访问
- 统一响应格式：`{ code: 0, data: {...} }`
- Webhook 支持外部系统集成
- 无复杂认证（MVP 阶段）

## 上下文信誉体系

### 查询信誉

```
GET /api/v1/reputation?userId=agent_001&context=towow

Response:
{
  "code": 0,
  "data": {
    "userId": "agent_001",
    "context": "towow",
    "fulfillmentRate": 0.95,
    "totalCommitments": 20,
    "fulfilledCount": 19,
    "failedCount": 1,
    "pendingCount": 0,
    "lastActive": "2026-02-23"
  }
}
```

### 不提供全局评分

Xenos 不计算"总体信誉分"，因为：

1. 不同上下文的承诺不可比
2. 全局评分会掩盖领域差异
3. 调用方应根据上下文自行决策

## 角色定义

| 角色 | 说明 |
|------|------|
| 承诺方 (Promiser) | 承诺完成任务的人（接任务） |
| 委托方 (Delegator) | 发布任务、等待完成的人（发任务） |
| 验收方 (Verifier) | 验收承诺是否完成（通常是委托方） |
